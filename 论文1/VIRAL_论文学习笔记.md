## 一句话概括

> VIRAL 证明了：通过大规模仿真中的 RL 训练 + 教师-学生蒸馏 + 充分的视觉域随机化，人形机器人可以仅靠机载 RGB 摄像头实现零样本 Sim-to-Real 运动操控（边走路边抓放物体），在真实世界中达到 91.5% 成功率，逼近人类专家遥操作水平。

## 二、核心问题与动机

### 2.1 领域现状

| 方向 | 当前水平 | 瓶颈 |
|------|---------|------|
| 🦿 **盲态足式运动** | 仿真训练 → 真机零样本部署 ✅ | 只管走路，不管手上活 |
| ✋ **桌面操作** | 以真机模仿学习为主 | 固定底座，不能移动 |
| 🧠 **机器人基础模型** | 大规模遥操作数据 → 训大模型 | 人形移动操作数据成本天价 |
| 🚶‍♂️+✋ **运动操控** | 几乎无人做到视觉 Sim-to-Real | 运动+操作耦合，难度乘积级上升 |

### 2.2 本文锚定的核心问题

> **"视觉 Sim-to-Real 能不能让仅靠机载摄像头的人形机器人，实现有用的运动操控？"**

### 2.3 本文定位

**不提出新的 RL 或 Sim-to-Real 算法**，而是提供一个**全栈技术配方（full-stack recipe）**，回答：要把 RGB 人形运动操控落地，哪些设计是必要的、哪些会失败、它们之间如何相互作用。

## 四、Phase 1：RL 教师训练（四个关键设计）

### 核心定义

> **教师 = 一个在仿真器中"开上帝视角"的 PPO 强化学习策略。** 输入是自己的全身状态 + 环境精确 3D 信息，输出给下层 WBC 的增量式高层指令，用分布式多 GPU 训练。

### 教师输入（特权观测）

```
观测 ot_priv = [ 本体感知 (proprio)  +  外部感知 (extero) ]
```

#### 本体感知 o_prop-priv —— "我自己的身体状态"

| 变量 | 含义 | 通俗理解 |
|------|------|----------|
| **v_t** | 基座线速度 | "身体在往哪飘、多快" |
| **ω_t** | 基座角速度 | "身体在转多快" |
| **g_t** | 重力投影 | "我朝哪边倾斜、倾了多少"——感知姿态的关键信号 |
| **a_{t-1}** | 上一时刻的动作 | "我刚刚做了什么"——提供时序连续性 |
| **q_t** | 所有关节的当前位置 | "每个关节现在多少度" |
| **q̇_t** | 所有关节的当前速度 | "每个关节转多快" |
| **f_finger** | 指尖力 | "手指有没有碰到东西、使了多大力" |

#### 外部感知 o_ext-priv —— "环境里的精确信息"

| 变量 | 含义 | 通俗理解 |
|------|------|----------|
| **e_t** | 当前任务阶段 | "现在该走、该放、还是该抓？" |
| **T_t** | 放置/抓取目标位置 | "物体要放到哪、要从哪拿" |
| **O_t** | 物体、桌子相对机器人的精确 3D 位姿 | "桌子在我左前方 1.2m，物体在桌上 0.3m" |

> ⚡ **关键洞察**：O_t（精确 3D 位姿）在真机上永远不可能精确获取——单目 RGB 相机只能拍二维图像，做不到毫米级精度的物体 6D 位姿估计。这就是"特权"二字的含义。

### 教师输出（动作空间）

```
a_t = (Δv, ω_yaw, Δq_arm, Δq_finger)    ← 全部是增量式 WBC 高层指令
```

| 输出 | 维度 | 含义 |
|------|------|------|
| Δv | 2 维 (x,y) | 身体前后左右速度增量 |
| ω_yaw | 1 维 | 绕竖直轴偏航角速度 |
| Δq_arm | 7 维 | 手臂各关节角度增量 |
| Δq_finger | 6 维 | 手指各关节角度增量 |

**为什么是增量（Delta）而非绝对值？** 详见 4.2 节。

---

### 4.1 教师要素①：奖励设计 —— 分阶段密集引导

#### 问题

长周期任务（走→抓→放→转→走→抓…）如果用单一稀疏奖励（如"抓到物体=1分"），RL 瞎碰一万次也碰不到成功状态——这就是探索瓶颈。

#### 解法：将完整任务拆成四个子阶段，每阶段独立设计密集奖励

```
任务序列: 走向物体 → 抓取物体 → 走向托盘 → 放置物体 → 转身 → (循环)
          └── ① ──┘ └── ③ ──┘          └── ② ──┘ └─ ④ ─┘
```

#### ① 走向物体 r_walk

```
r_walk = exp( -4 × (距离 - 0.45)² )

距离 = |机器人位置 - 要抓的物体位置|
```

| 特性 | 说明 |
|------|------|
| 最佳距离 | 0.45m（伸手刚好够到的距离） |
| 函数形状 | 指数衰减，越接近理想距离奖励增长越快 |
| 梯度特性 | 清晰，RL 容易沿着梯度优化 |

#### ② 放置物体 r_place

```
r_place = f_PlaceObj × 𝟙(手-托盘距离 < 0.3m)
```

| 特性 | 说明 |
|------|------|
| f_PlaceObj | 手指和物体之间的接触力 |
| 激活条件 | 仅当手离托盘 < 30cm 时奖励才生效 |
| 设计意图 | 鼓励"靠近托盘时松手"——离远了松手没有奖励，策略自然学会时机 |

#### ③ 抓取物体 r_grasp（两个子奖励）

```
r_grasp-z  = min(物体高度 - 桌面高度, 0.15)    ← 提离桌面，15cm 封顶
r_grasp-goal = exp( -10 × ||物体位置 - 目标位置||² )  ← 精准对准目标
```

| 特性 | 说明 |
|------|------|
| r_grasp-z | 鼓励把物体提起来，15cm 后不再奖励——防止手臂乱挥 |
| r_grasp-goal | 指数系数 -10 很陡，差一点奖励就降很多——鼓励精准放置 |

#### ④ 转身 r_turn

```
r_turn = -|机器人当前朝向 - 目标朝向|     ← 负绝对值，偏差越大惩罚越重
```

| 特性 | 说明 |
|------|------|
| 零偏差时 | 奖励=0（最大） |
| 设计意图 | 完成任务后转身面向下一个目标，最简单的方向对齐信号 |

#### 奖励设计的核心启示

> 长周期 RL 任务不要试图用一个统一的奖励函数——**先分解任务阶段，再针对每个阶段设计密集引导奖励**，让策略在每一小步都有清晰的优化方向。

### 4.2 教师要素②：增量动作空间 —— Δ 优于绝对值

#### 两种动作空间对比

| | 绝对动作空间 (Absolute) | 增量动作空间 (Delta) |
|---|---|---|
| **输出** | `q_target = 45°` | `Δq = +3°` |
| **执行方式** | 直接跳到目标角度 | 在上一拍基础上累加 |
| **平滑性** | 可能跳变，帧间不连续 | 天然平滑，连续变化 |
| **RL 探索难度** | 高——状态-动作映射空间巨大 | 低——搜索范围限定在"微调" |
| **适合场景** | 周期性动作（纯走路） | 多阶段、非周期操作（运动操控） |

#### 为什么在运动操控中 Delta 更优？

```
绝对动作在同一姿态下："右肘=45°"
  站着时 "右肘=45°" 可能手在胸前 →
  蹲着时 "右肘=45°" 可能手在头顶 → 同一个数值在不同姿态下含义完全不同！

增量动作："在上一次基础上 +3°"
  不管当前姿态如何，+3° 永远是微调 → 搜索空间小且稳定
```

#### 消融结论

> **绝对动作空间几乎学不会该任务（成功率趋近于 0），增量动作空间成功率 ~95%。** 这是人形运动操控与纯走路 RL 的一个关键设计差异。

### 4.3 教师要素③：WBC 作为 API 层 —— 不造轮子，调用现成能力

#### 从零学力矩控制的困境

如果让 RL 直接输出 29 个关节的电机力矩，你需要设计奖励覆盖：走路速度、摔倒、关节限位、脚底打滑、身体晃动、手臂-腿自碰撞、能耗、抬脚时机……几十项，而且仿真完美但真机一跑就摔。

#### VIRAL 的解法

```
传统端到端 RL:                       VIRAL 做法:

  RL 策略                              RL 策略 (VIRAL Teacher)
    ↓                                     ↓
  输出：29 个关节力矩                   输出：6 维 WBC 命令
    ↓                                     ↓
  物理引擎直接模拟                   ┌── HOMIE WBC 策略 ──┐
                                    │ · 稳定行走          │
                                    │ · 全身平衡          │
                                    │ · 关节协调          │
                                    │ · 输出力矩到 PD 控制 │
                                    └────────────────────┘
```

#### 类比理解

| | 造车 | 开车 |
|---|---|---|
| **做什么** | 设计发动机、变速箱、悬挂、刹车 | 转方向盘 + 踩油门/刹车 |
| **对应** | HOMIE 做的事（底层控制） | VIRAL 做的事（高层决策） |
| **类比** | 写操作系统内核 | 调用操作系统 API |

> VIRAL 的贡献不是重新造 HOMIE，而是**教会 AI 如何"开车"**——用 RGB 视觉判断路况、自主决定方向盘和油门。HOMIE 负责保证车本身能稳定运行。

#### HOMIE 简介

| 特性 | 说明 |
|------|------|
| 全称 | Humanoid Loco-Manipulation with Isomorphic Exoskeleton Cockpit |
| 发表 | RSS 2025（上海 AI Lab × CUHK） |
| 训练方式 | RL 训练，不依赖动捕数据 |
| 核心创新 | 上半身姿态课程学习、高度跟踪奖励、对称性利用 |
| 命令接口 | 速度 + 高度 + 上半身关节 +（VIRAL 扩展了手指） |

### 4.4 教师要素④：参考状态初始化 RSI —— 打破探索瓶颈

#### 问题：长周期任务的顺序探索瓶颈

从零开始的 RL：先学会站（10 小时）→ 学会走（50 小时）→ 走到桌边（200 小时）→ 碰巧抓住物体……每向前突破一步，**必须先稳定掌握前置步骤**，后续阶段永远得不到充分练习。

#### RSI 的做法

```
1. 人类遥操作专家在仿真中完整执行任务 200 次 → 录制所有状态快照
2. 每次 RL episode 重置时：
   · 随机从某次演示中采样一个中间状态
   · 将机器人、物体、桌子全部"瞬移"到该状态
   · RL 策略从这个有帮助的位置继续
```

#### 效果对比

```
传统 RL（无 RSI）:
  Episode 以 "机器人傻站在原点" 开始 → 99% 时间在练走路 → 几乎不练抓取
  → 成功率 < 10%

RSI:
  Episode 可能直接从 "手离物体 10cm" 开始 → 大量练习抓取
  Episode 可能直接从 "抓住物体站着" 开始 → 大量练习走回托盘
  → 成功率 ~95%
```

#### 学习启示

> RSI 的核心思想是**在 RL 训练中人为注入"好状态"的初始分布**，让策略在每个任务阶段都得到充分练习，而不是被前置步骤卡死。这是一种介于纯 RL 和模仿学习之间的混合策略——利用少量演示来引导 RL 的探索方向。

## 五、Phase 2：视觉学生蒸馏（三个关键设计）

#### 核心定义

> **蒸馏 = 把 RL 问题转化为监督学习问题。** 教师通过 PPO 学会了"看到 X 该做 Y"，学生只需要监督学习模仿这个映射——但学生只能看到 RGB 图像和本体感知，拿不到教师的特权信息。

---

### 5.1 学生要素①：DAgger + BC 混合 —— 标准答案 + 改错本

#### 三种蒸馏策略的本质

```
                    ┌──────────────────────────────────────────────┐
                    │              数据收集方式                      │
                    ├──────────────────┬───────────────────────────┤
  BC (λ=1)          │ Teacher 自己跑     │ 只看完美示范               │
                    │ 学生被动学         │ 不会纠错，部署即崩溃         │
                    ├──────────────────┼───────────────────────────┤
  DAgger (λ=0)      │ Student 自己跑     │ 覆盖错误状态               │
                    │ Teacher 批改       │ 鲁棒但收敛慢               │
                    ├──────────────────┼───────────────────────────┤
  混合 (λ=0.5) ✅   │ 50% BC + 50% DAgger│ 既有标准答案，又有改错本      │
                    └──────────────────┴───────────────────────────┘
```

#### 训练过程

```
每轮数据收集:
  · 50% 仿真环境 → Teacher 自己跑（产生 BC 数据：完美轨迹）
  · 50% 仿真环境 → Student 跑，Teacher 对每帧批改（产生 DAgger 数据：纠错样本）
  · 两批数据混在一起 → MSE(学生动作, 教师动作) → 更新学生参数
```

#### 为什么纯 BC 不行？—— 分布漂移（Distribution Shift）

```
纯 BC 的致命问题:
  Teacher 的轨迹永远完美 → 训练数据中只有"正确路径"上的状态
  Student 一旦偏了一点 → 进入训练数据中不存在的"陌生状态"
  → 不会应对 → 乱猜 → 偏得更多 → 彻底崩溃（复合误差 compounding error）

DAgger 的修正:
  Student 自己跑 → 天然会产生各种"偏了的状态"
  Teacher 对着这些状态批改 → 训练数据覆盖了犯错区域
  → 学生学会纠错 → 部署鲁棒
```

#### 消融结论

> **λ=0.5 的混合策略最优。** 纯 BC 真机成功率低（策略脆弱），纯 DAgger 收敛太慢，混合兼顾了收敛速度和部署鲁棒性。

### 5.2 学生要素②：网络结构 —— DINOv3 + LSTM + 本体感知拼接

```
                        ┌──────────────────────────────────┐
                        │        Student 完整网络结构        │
                        └──────────────────────────────────┘

   输入①: RGB 图像                            输入②: 本体感知
   (480×640×3, 单目)                          (~50 维向量)
         │                                         │
         ▼                                         │
   ┌──────────────┐                                │
   │   DINOv3     │  ← 视觉骨干（自监督预训练）        │
   │   输出: 768维 │                                │
   │   视觉特征    │  ← 从像素中推断出的外部感知近似     │
   └──────┬───────┘                                │
          │                                        │
          └────────────┬───────────────────────────┘
                       ▼
              ┌─────────────────┐
              │  concat 拼接     │
              │  [768 + 50] 维  │
              └────────┬────────┘
                       ▼
              ┌─────────────────┐
              │     LSTM        │  ← 时序建模，维护隐状态 h_t
              │  输入: 拼接向量  │     "选择性记忆"历史关键信息
              │  + h_{t-1}      │
              └────────┬────────┘
                       ▼
              ┌─────────────────┐
              │   MLP 输出头     │
              └────────┬────────┘
                       ▼
                  a_student
          (Δv, ω_yaw, Δq_arm, Δq_finger)
```

#### 三种历史建模方式

| | 单步 MLP | 前馈历史 | LSTM ✅ |
|---|---|---|---|
| **"记忆"形式** | 无 | 固定窗口拼接 | 可学习隐状态，门控更新 |
| **能看多远** | 只看当前帧 | 固定 N 帧 | 理论上无限长 |
| **输入维度** | 818 | 818 × N（膨胀） | 818（固定） |
| **适合场景** | 反应式任务 | 短期时序 | **长周期、需上下文的任务** |
| **VIRAL 表现** | 最差 | 中等 | 最优 |

#### LSTM 中的"隐状态"到底是什么？

> **隐状态 h_t 不是"上一帧的观测的拷贝"，而是网络自己选择性记下来的历史摘要。** 三个门控（遗忘门/输入门/输出门）决定：哪些旧信息该忘、哪些新信息该记、现在该输出什么。叫"隐"（hidden）是因为这个摘要的内容你永远看不见，也从来不需要你手动标注——是网络在训练中自己学会怎么记的。

#### 视觉特征 ≈ 外部感知的神经网络近似

```
教师的外部感知（特权）:          学生的视觉特征（从像素学出的替代）:
  "桌子在 (1.2m, 0.3m, 0.75m)"    DINOv3 输出 768 维向量，某些维度
  精确 3D 坐标，程序直接读取        的高激活隐含编码了"桌子在左前方"
  
  → 精确但真机永远拿不到            → 有噪声但真机上完全可获取
```

### 5.3 学生要素③：分布式训练 —— 算力是关键

#### 为什么需要分布式

```
教师训练（无渲染）:  1 GPU → ~10k steps/s
学生训练（需渲染）:  1 GPU → ~500 steps/s    ← 慢 20 倍！
                    + DINOv3 前向推理更慢
```

#### 分布式训练 ≠ 各训各的

| 你以为的堆 GPU | 实际的分布式训练 |
|--------------|----------------|
| 64 张卡各跑各的 | 64 张卡**共享同一套策略参数** |
| 数据不互通 | 每轮 All-Gather 汇总所有环境的数据 |
| 各更新各的 | 统一算梯度 → 广播同步 → 进入下一轮 |

#### 并行粒度的影响

| GPU 数 | 效果 |
|--------|------|
| 1 GPU | 收敛极慢，成功率低，训练曲线剧烈抖动 |
| 8 GPU | 能收敛但不稳定 |
| 16 GPU | 成功率明显提升 |
| **64 GPU** | 最快收敛，最高成功率，训练曲线平滑如丝 |

> **规模效应的两个关键发现**：
> ① 不止加速，更是质量提升——每轮参数更新时模型看到了更多样的场景（256 个并行环境同时有亮/暗/不同桌子/不同物体）
> ② 大 batch → 梯度估计更准 → 训练更稳定

#### 技术栈

```
TRL (PPO 算法逻辑)   +   Accelerate (多GPU通信)   +   Isaac Lab (仿真管理)
        │                         │                          │
        └─────────────────────────┼──────────────────────────┘
                                  ▼
                   自定义分布式 PPO 训练框架:
                   · TRL 提供 PPO 算法实现（rollout/GAE/clip/更新）
                   · Accelerate 提供 All-Gather/梯度同步/广播
                   · 支持 1 GPU → 64 GPU 线性扩展
```

## 七、概念深度辨析

### 7.1 蒸馏到底是什么？

> **"老师教学生"** —— 先让能作弊的教师（访问仿真器全部状态）通过 RL 学会任务，再让学生只看教师的输入输出对来模仿。学生输入的是真机可获取的信息（RGB + 关节状态），输出的是教师的标准动作。一旦蒸馏完成，学生可以零样本上真机——因为它的所有输入真机都拿得到。

### 7.2 BC vs DAgger vs 混合

| | BC | DAgger | 混合 (λ=0.5) |
|---|---|---|---|
| **数据来源** | 只听教师的 | 学生自己跑，教师批改 | 各占一半 |
| **优点** | 收敛快，loss 降得快 | 覆盖错误状态，鲁棒 | 两者兼顾 ✅ |
| **致命问题** | 分布漂移 → 上线崩溃 | 收敛太慢 | — |
| **本质** | 标准答案死记硬背 | 边犯错边改错 | 打基础 + 做改错本 |

### 7.3 WBC（全身控制）详解

> **WBC 是一种控制架构思想**——把全身所有关节放在一起协调控制，保证走路和动手不冲突。VLA（Vision-Language-Action）把 WBC 训好的策略当作现成的 API 层调用，VIRAL 只输出高层速度/关节命令，由 WBC 负责底层力矩执行。WBC 本身也可以是用 RL 训练的（如 HOMIE），这是一个层级 RL 架构。

### 7.4 监督学习 vs 强化学习 vs 无监督学习

| | 监督学习 | 强化学习 | 无监督学习 |
|---|---|---|---|
| **数据形式** | (输入, 标签) 对 | (状态, 动作, 奖励, 下一状态) | 只有输入，无标签 |
| **有标准答案吗** | ✅ 有 | ❌ 只有奖励信号 | ❌ 没有 |
| **怎么学** | 模仿答案 | 试错 → 得奖励 → 调整策略 | 自己找数据中的规律 |
| **典型例子** | 图像分类、语音识别 | AlphaGo、机器人控制 | 聚类、降维、自编码 |
| **VIRAL 中的应用** | Phase 2 学生蒸馏 | Phase 1 教师训练 | — |

### 7.5 单步 MLP vs 前馈历史 vs LSTM

```
单步 MLP:   [当前帧] ──→ MLP ──→ 动作    ← "只看现在，不记得过去"
前馈历史:   [t, t-1, t-2, t-3, t-4] ──→ MLP ──→ 动作   ← "看完5张照片再决定"
LSTM:       [当前帧 + h_{t-1}] ──→ LSTM ──→ [动作, h_t]   ← "自己写笔记，边记边忘"
                                                                   ↑
                                                           h_t 是历史的可学习摘要
```

| 核心差异 | 前馈历史 | LSTM |
|---------|---------|------|
| **历史存储** | 原始数据拼接在输入里 | 压缩在学习好的隐状态里 |
| **可记忆长度** | 固定窗口（超参数） | 理论无限长 |
| **遗忘机制** | 无——窗口外的信息直接丢失 | 有——遗忘门主动丢弃不重要信息 |
| **参数效率** | 输入维度随窗口线性增长 | 固定参数量 |

### 7.6 分布式训练 ≠ 堆 GPU

> **分布式训练 = 数据并行 + 参数同步。** 多 GPU 共用同一本"施工手册"（策略参数），各自在环境中采集经验，汇总后统一更新手册，再广播给所有人。TRL 提供 PPO 的算法实现，Accelerate 提供 GPU 间的通信原语（All-Gather/梯度同步），两者配合实现线性扩展。
